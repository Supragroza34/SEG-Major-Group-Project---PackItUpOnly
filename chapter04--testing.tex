\section{Testing approach}

Our testing strategy combined \textbf{automated testing} (unit, integration, and API/UI behaviour tests) with targeted \textbf{manual testing} for user experience and end-to-end acceptance checks.

The approach followed three principles:
\begin{enumerate}
	\item \textbf{Risk-first coverage:} prioritise workflows that affect core stakeholder value (ticket creation, dashboard visibility, status updates, replies, meeting requests, and admin controls).
	\item \textbf{Fast feedback:} run backend and frontend test suites locally via scripted commands before merging.
	\item \textbf{Defect containment:} use regression tests when bugs are found so the same issue is less likely to recur.
\end{enumerate}

\section{Automated testing}

\subsection{Suite scope and structure}

The delivered codebase contains a broad automated suite across backend and frontend:
\begin{itemize}
	\item \textbf{Backend (Django/DRF):} 39 test files and 594 test cases.
	\item \textbf{Frontend (React/Jest):} 42 test files and 354 test cases.
	\item \textbf{Total:} 81 test files and 948 automated test cases.
\end{itemize}

Backend tests are concentrated in \texttt{KCLTicketingSystems/tests/}, with additional configuration tests in \texttt{KCLTicketingSystem/tests/} and chatbot tests in \texttt{AIChatbot/tests.py}. Frontend tests are distributed across pages, components, API service modules, utility functions, and Redux slices.

\subsection{Backend testing (unit and integration)}

Backend tests cover:
\begin{itemize}
	\item \textbf{Model-level unit tests:} validation, field constraints, timestamps, relationships, and string representations.
	\item \textbf{API integration tests:} request/response behaviour, authentication (JWT), permission boundaries, role-based access, and error handling.
	\item \textbf{Feature/regression tests:} ticket conversation, rich-text handling, notification behaviour, PDF export, admin reporting, filtering, and staff assignment logic.
\end{itemize}

\subsection{Frontend testing (component and behaviour)}

Frontend tests use React Testing Library and Jest to verify:
\begin{itemize}
	\item \textbf{Rendering and interaction:} pages, components, and conditional UI states.
	\item \textbf{Form behaviour:} validation, field updates, submission success/failure paths.
	\item \textbf{Routing and access control:} protected routes and role-based navigation.
	\item \textbf{State management:} Redux slice reducers and async thunk outcomes.
	\item \textbf{API client behaviour:} request headers, payload handling, and error mapping.
\end{itemize}

\subsection{Execution pipeline}

To keep execution simple for all team members, test orchestration scripts were provided:
\begin{itemize}
	\item \texttt{run\_all\_tests.bat} (Windows)
	\item \texttt{run\_all\_tests.sh} (Linux/macOS)
\end{itemize}

These run backend tests first, then frontend tests, giving a single pass/fail checkpoint.

On the most recent full backend run in this branch:
\begin{itemize}
	\item \textbf{574 backend tests} were discovered and executed.
	\item \textbf{569 passed}, with \textbf{1 failure} and \textbf{4 errors} detected.

\section{Manual testing}

\subsection{Purpose of manual testing}

Manual testing was used primarily for areas where \emph{human judgement} is essential:
\begin{itemize}
	\item visual consistency and usability of multi-step user journeys;
	\item perceived clarity of error/success messaging;
	\item role-specific UX checks (student vs. staff vs. admin);
	\item exploratory checks after UI or workflow changes.
\end{itemize}

\subsection{Extent of reliance on manual testing}

The team relied mainly on automated tests for repeatable correctness and regression control. Manual testing was complementary rather than primary. In practice:
\begin{itemize}
	\item \textbf{Automated tests} handled most deterministic validation (input rules, permissions, response codes, state transitions).
	\item \textbf{Manual tests} were used for end-to-end confidence in user-facing behaviour and presentation quality.
\end{itemize}

\subsection{Limits of automated and manual testing}

	extbf{Automated testing limits:}
\begin{itemize}
	\item may miss usability/accessibility perception issues;
	\item can become brittle when UI markup changes;
	\item does not fully represent all browser/device behaviours unless explicitly expanded.
\end{itemize}

	extbf{Manual testing limits:}
\begin{itemize}
	\item slower and harder to repeat at high frequency;
	\item vulnerable to inconsistent execution between testers;
	\item weaker as a long-term regression safety net.
\end{itemize}

\section{Summary}

Overall, the project uses a \textbf{hybrid testing strategy} with strong automated regression coverage and focused manual validation for usability and end-user confidence. The breadth of the automated suite (948 test cases) provides a solid technical baseline, while manual checks ensure the delivered product remains usable and coherent for students, staff, and administrators.
